## ISSUE 1 — Delete order with X on `/account/orders`

- **What user said:** "view order, delete with x"
- **Audit finding:** `app/(shop)/account/orders/page.tsx` lines 113–184 renders order cards. No delete affordance. Backend has no soft-delete column on `orders` (see `supabase/migrations/003_orders.sql` lines 3–30).
- **Decision (locked):** Only cancelled/unpaid orders are deletable. Soft-hide from customer view only — admin still sees them.
- **Plan:**
  - DB: new migration `025_orders_customer_hidden.sql` — `ALTER TABLE orders ADD COLUMN customer_hidden_at TIMESTAMPTZ NULL`. Index on `(customer_email, customer_hidden_at)`.
  - API: `app/api/orders/hide/route.ts` — POST `{ order_number }`. Auth required (session email must match `customer_email`). Reject if `payment_status === 'paid'` AND `fulfillment_status NOT IN ('cancelled','unfulfilled')`. Set `customer_hidden_at = NOW()`.
  - Page: `app/(shop)/account/orders/page.tsx` — getOrders() adds `.is('customer_hidden_at', null)`. Each card gets X button top-right (only render if eligible). Confirm modal before delete. Optimistic UI + revalidate via server action.
  - Component: new `components/account/OrderCardActions.tsx` (client) — X button + confirm dialog.
- **Acceptance:**
  - X visible only when `payment_status IN ('pending','failed','refunded')` OR `fulfillment_status === 'cancelled'`.
  - Click X → confirm modal "Hide order #XXXXX?" → on confirm row hidden from `/account/orders`.
  - Admin `/admin/orders` still shows the order.
  - Reload still hides it (persisted).
  - Unauthorised hide attempt (different user's order_number) returns 403.
- **Risk:** None significant. Soft-hide is reversible.

## ISSUE 3 — Save address, prefill checkout second time

- **What user said:** "save info like address one and keep it generic and easy checkout second time"
- **Audit finding:** `user_addresses` table already exists (`supabase/migrations/006_user_addresses.sql`) with `is_default` flag. **Not wired into checkout.** Checkout form (`app/(shop)/checkout/CheckoutForm.tsx`) collects address fresh every time, never reads or writes `user_addresses`.
- **Decision (locked):** Full address book. Auto-save first checkout as default. Prefill next time. Add/edit/delete in account. Pick from saved list at checkout.
- **Plan:**
  - **DB:** existing `user_addresses` is fine. Add unique partial index: `CREATE UNIQUE INDEX idx_one_default_per_user ON user_addresses(user_id) WHERE is_default = TRUE` to enforce single default. Migration `026_addresses_default_unique.sql`.
  - **API endpoints** (all under `app/api/account/addresses/`):
    - `GET /api/account/addresses` — list current user's addresses, default first.
    - `POST /api/account/addresses` — create. If `is_default` true OR first address, unset others' defaults atomically.
    - `PATCH /api/account/addresses/[id]` — update fields, can toggle default.
    - `DELETE /api/account/addresses/[id]` — delete (if default, promote oldest remaining to default).
  - **Account UI:** `app/(shop)/account/addresses/page.tsx` — list cards + add/edit modal. Link from `/account` index.
  - **Checkout:** `app/(shop)/checkout/CheckoutForm.tsx`:
    - On mount, if user logged in, fetch addresses. If any exist → show "Use saved address" picker dropdown above the form fields (collapsed list with default selected).
    - "Add new address" option in dropdown → expand blank form.
    - If only one default → autofill silently, show "Edit address" link.
    - On successful order (after Razorpay verify), if address was newly entered, POST to `/api/account/addresses` with `is_default: !hasAnyAddress`. Silent; no extra prompt.
  - **Dev session fallback** (currently `dev@thepossah.com` hardcoded in orders page line 52) — same fallback in checkout API so dev mode works.
- **Acceptance:**
  - Logged-in user with 0 saved addresses checks out → address saved as default automatically.
  - Second checkout → address prefilled, can pick from dropdown or add new.
  - `/account/addresses` lists all, marks default, allows add/edit/delete.
  - Deleting default promotes oldest remaining to default.
  - Two defaults impossible (unique index).
- **Risk:** Existing checkout flow is battle-tested. Don't break the unauthenticated guest path (guests just type address, no save). Feature-flag the prefill if needed.
